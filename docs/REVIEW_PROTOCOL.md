# 评审协议 v1(契约式互审)

> 当前版本:**A 级 - 契约式互审**。仅在 Agent 间存在接口契约时强制评审,不做对抗式评审和跨域强制评审。后续按需升级。

## 一、核心原则

1. **产出方写完不算完,消费方签字才算完**
2. **评审必须给具体修改建议,不接受"看起来不行"这种反馈**
3. **每个产出最多 2 轮评审,第 3 轮分歧由 Lead 仲裁并记入 ADR**
4. **评审不阻塞探索,只阻塞合并**:Sub-agent 可以并行做实验性产出,正式进入下游前才走评审

## 二、评审契约矩阵

下表定义了哪些产出必须由谁签字。**列出的关系是强制的,未列出的关系靠 Lead 判断**。

| 产出类型 | 责任 Agent | 评审 Agent | 关注点 |
|---------|-----------|-----------|--------|
| PRD / 验收标准 | pm | qa | 验收标准是否可测试、Given-When-Then 是否完整 |
| 设计规范 / 组件清单 | ux | frontend | 组件是否可实现、Token 是否齐全、双端差异是否标注 |
| OpenAPI 契约 | backend | frontend | 字段是否够用、错误码是否覆盖、分页/排序是否合理 |
| OpenAPI 契约 | backend | ai-integration(若涉及 AI 接口) | AI 调用语义是否清晰、超时/降级字段是否预留 |
| 数据库迁移脚本 | backend | devops | 是否可平滑迁移、是否有回滚脚本、是否锁表 |
| Skill / Prompt 新增或重大改动 | ai-integration | qa | 是否有测试集、是否覆盖 bad case |
| 埋点 Schema | analytics | frontend / backend(谁上报谁评审) | 字段命名、上报时机是否合理、是否带 tenant_id |
| Helm Chart / 部署配置 | devops | backend | 资源配额、健康检查、配置项是否完整 |
| 灰度策略 | devops | qa | 灰度阶段的验证标准是否清晰 |

## 三、评审签字流程

### 标准流程
```
1. 产出方写完产出物(代码 / 文档 / 配置)
2. 产出方在 tasks/TASK-{id}.md 的"评审签字区"勾选自己的"已就绪"
3. 产出方在签字区 @ 评审方,留下产出物路径
4. 评审方读产出物,在签字区写评审意见
5. 评审通过 → 评审方勾"通过";不通过 → 留具体修改建议,产出方修改后重新走流程
6. 第 2 轮评审仍不通过 → 升级到 Lead 仲裁
```

### 评审意见格式

```markdown
### {评审方 Agent} 的评审意见 - 第 N 轮

**结论**: ☐ 通过  ☐ 通过(附建议)  ☐ 打回

**问题清单**:
1. [具体问题 + 涉及文件/行号 + 修改建议]
2. ...

**风险提示**(若有):
- ...
```

### 评审超时
评审方收到评审请求后,默认 1 个工作时段内响应。超时由 Lead 介入催办或代审。

## 四、Lead 的角色

Lead **不直接做契约式评审**(那是 Agent 间的事),Lead 的职责:
1. 检查每个进入合并阶段的任务是否签字齐全
2. 第 2 轮评审仍分歧时介入仲裁
3. 仲裁结果写入 `docs/adr/ADR-{date}-{slug}.md`
4. 维护本协议:发现新的契约关系应该补入矩阵

## 五、不做评审的情况(明确白名单)

为了避免流程化,以下情况**不强制**走签字:
- 产出方 < 30 行代码且不涉及契约的改动(纯内部重构)
- 紧急 P0 修复(事后补 Postmortem)
- 文档改动(typo / 注释)
- 仅本 Agent 内部的工具脚本

## 六、与 PR 流程的衔接

```
代码就绪
   ↓
任务文档评审签字齐全(本协议)
   ↓
QA 准入 checklist 全过
   ↓
Lead 架构审查
   ↓
合并主干
```

签字未齐全 → QA Agent 直接打回,不进入准入检查。

## 七、升级到 B 的触发条件(参考)

满足任一条建议升级:
- 一个月内 ≥ 2 次"该被发现的"线上 Bug
- 跨服务改动 ≥ 每周 2 次
- 同时在跑特性 ≥ 3 个,Lead 审不过来
- 出现 Agent 间"踢皮球"现象

升级 B 增加:对抗式评审(红队模式)+ 跨域强制评审矩阵。
